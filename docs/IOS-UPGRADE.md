## Audit — what the app actually is

- **UI stack:** SwiftUI throughout (no UIKit view controllers), iOS 17
  minimum, Swift 6 with strict concurrency, Observation-based MVVM
  (`@Observable` feature models), zero third-party runtime dependencies.
  Project is generated by XcodeGen (`ios/project.yml`).
- **Authentication:** opaque rotating bearer sessions against the mounted
  `/api/v1` FastAPI sub-app (ADR 0066) — access ~15 min, refresh rotates
  with family-revocation on reuse. Session lives in a ThisDeviceOnly,
  origin-scoped Keychain item (`KeychainSessionPersistence`); refresh is
  actor-serialized (`SessionAuthenticator`); Face ID/Touch ID is a local
  re-entry lock only, never server auth. Cookies are never sent; redirects
  on API calls are rejected outright (`RedirectRejectingDelegate`).
- **API surface called:** tenant discovery, studio login, the four
  client-capability unlocks, refresh/logout/me/session-revocation;
  owner reads (dashboard, clients, projects, galleries + manifest,
  event-types, bookings); client reads (`/client/home`,
  `/client/galleries[/{id}]`, `/client/bookings`,
  `/projects/{id}/proposals|contracts|invoices`); the favorite
  PUT/DELETE mutation; and bearer-authenticated media
  (`/api/v1/media/...` via `AuthenticatedMediaLoader`).
- **AI exposure: none.** The app talks only to the Mise backend. No AI
  capability, sidecar, or model endpoint is reachable from the app — all
  AI surfaces remain behind the backend's provider facade. This matches
  the architecture directive and should be preserved as an invariant.
- **State of the milestones** (`docs/IOS-ARCHITECTURE.md` §10): M0–M3
  implemented and merged (M3 = client delivery, PR #144, today). M4 (safe
  mutations) and M5 (operations: APNs, deep links, telemetry, TestFlight)
  are planned but unstarted.
- **Known soft spots:** Swift unit tests exist (12 files) but **no CI job
  builds or runs them** — `.github/workflows/ci.yml` has no macOS runner,
  so iOS regressions are invisible until someone opens Xcode. Display
  typography uses the system serif as a stand-in for the design handoff's
  Newsreader (bundling is a pending licensing/asset decision). Video
  playback, comments, and background downloads are deferred from M3.
  `OwnerResourceModel`/`OwnerResourceView` are generic but owner-named.

## Open product question — do not assume

**Does this app serve kleephotography's single-tenant production use only,
or does it need eventual awareness of hosted SaaS multi-tenant mode?**
Today the app is tenant-agnostic in the right way (workspace = one origin,
host-first tenant resolution, per-origin Keychain isolation), and
`ios/README.md` treats hosted vs self-hosted as deliberately separate.
Distribution is the fork: a single-studio internal app (unlisted/TestFlight)
has different App Store, branding, and `MiseServerBaseURL` implications than
a multi-studio product where any studio's clients sign in. **Kevin decides;
nothing below depends on the answer until step 6.**

## Upgrade plan — in order

Each step is independently shippable; order minimizes rework.

1. **iOS CI gate (do first — everything after depends on it).** Add a
   macOS GitHub Actions job: `xcodegen generate` + `xcodebuild test` on an
   iOS 17 simulator, triggered on `ios/**` changes. Reasoning: three
   milestones of Swift shipped from Linux sessions with tests that have
   never run in CI; every later step lands blind without this. (Includes
   verifying the M3 test files compile — first CI run may need small
   fixes; that's the point.)
2. **Milestone 4a — low-risk owner mutations.** Task check-off and booking
   cancel/reschedule follow each existing `IOS-API-V1.md` command's
   idempotency contract. Task completion and booking cancellation are naturally
   idempotent without a key; reschedule requires one UUID `Idempotency-Key`
   preserved across transport retries. Show reschedule only when
   `available_commands` contains `booking.reschedule`; treat its `200` as
   booking-changed/workflow-queued, then follow the returned workflow status.
   Keep transitions server-authoritative and use optimistic UI only where the
   queued operation has a stable id. Start here because these are operator-only
   writes with no money/legal surface. Human review still gates the
   client-facing delivery activation.
3. **Commercial spine on mobile (the biggest product win —
   `MISE-REVIEW.md` §5).** New read-only `/api/v1` endpoints + owner
   screens for: AR chase assist (past-due invoices with chase state),
   closeout-readiness panel, company next-action ranking, Studio Activity
   commercial queue. Deterministic reads first (mirroring the admin
   panels), assist-style actions (e.g. "mark chased") as a follow-up under
   M4 rules. Reasoning: this is the owner's highest-value daily loop and
   it's currently desktop-only; reads are cheap and safe to ship.
4. **Milestone 4b — client decisions in-app, still web-executed.**
   Keep Accept/Sign/Pay on the canonical web pages (per ADR 0067), but add
   in-app `SFSafariViewController`/`ASWebAuthenticationSession` presentation
   plus post-return refresh, so the client never leaves the app shell.
   Native e-sign/checkout remain separate, legally-reviewed milestones.
5. **Milestone 5 — operations.** APNs device registry (`POST
   /api/v1/devices` per IOS-ARCHITECTURE §7), notification outbox,
   universal links (`/app/...` routes) resolved through the existing
   authorization-aware router, then TestFlight + privacy artifacts.
6. **Distribution decision (blocked on the product question above).**
   Single-tenant: pin branding, unlisted distribution, simplest path.
   Multi-tenant: keep the workspace-entry flow primary, add the App Store
   review account/demo studio, revisit `MiseServerBaseURL` provisioning.
7. **Polish debts, batched:** bundle Newsreader/Archivo once licensing is
   settled (OFL — verify embedding terms), rename
   `OwnerResourceModel/View` → `ResourceModel/View`, video playback via
   `AVPlayer` against the Range-capable media routes, gallery background
   downloads, document-level deep links from client Home next-steps.

Steps 2–5 map onto the existing milestone plan; this document sequences
them and inserts the commercial spine (step 3), which the milestone plan
predates. Anything touching money, auth, migrations, or contracts ships as
a reviewed draft PR per CLAUDE.md — flagged per-task in
`docs/HANDOFF-QUEUE.md`.
